# NuGet Packages

This guide explains how NuGet packages work in .NET development, particularly how they compare to pip packages in Python and how System-Response will be distributed as a NuGet package.

---

## What Is NuGet?

NuGet is the package manager for .NET, equivalent to pip for Python. It handles downloading, installing, and managing library dependencies.

## Project Reference vs Package Reference

There are two ways to reference another library in .NET:

### Project Reference (During Development)

When you have the source code locally and want to work on both projects together:

```xml
<!-- In RMCToolboxes.API.csproj -->
<ItemGroup>
    <!-- Points to local source code -->
    <ProjectReference Include="..\..\..\..\System-Response\SystemResponse\SystemResponse.csproj" />
</ItemGroup>
```

**Python equivalent**: Adding a local folder to your Python path:

```python
import sys
sys.path.append("C:/GitHub/System-Response")
from system_response import SchmertmannModel
```

### Package Reference (For Published Packages)

When using a published NuGet package:

```xml
<!-- In RMCToolboxes.API.csproj -->
<ItemGroup>
    <!-- Downloads from NuGet server -->
    <PackageReference Include="SystemResponse" Version="1.0.0" />
</ItemGroup>
```

**Python equivalent**: Installing from pip:

```bash
pip install system-response
```

### Your Code Stays Identical

The actual C# code you write does not change at all:

```csharp
// This code works EXACTLY the same whether System-Response is:
// - A project reference (local source code)
// - A NuGet package (downloaded binary)

using SystemResponse.Toolboxes.BEPProgression.Schmertmann;

public class SchmertmannController : ControllerBase
{
    [HttpPost("calculate/{analysisId}")]
    public async Task<IActionResult> Calculate(int analysisId)
    {
        var config = await LoadConfigFromDatabase(analysisId);

        // This line is IDENTICAL regardless of reference method
        var model = new SchmertmannModel(config);
        var result = await model.RunAsync();

        return Ok(result);
    }
}
```

## Working with NuGet

### Adding a Package

```bash
# From command line
dotnet add package SystemResponse --version 1.0.0

# Or edit .csproj directly
<PackageReference Include="SystemResponse" Version="1.0.0" />
```

### Updating a Package

```bash
# Update to latest version
dotnet add package SystemResponse

# Update to specific version
dotnet add package SystemResponse --version 1.1.0
```

### Restoring Packages

When you clone a project, NuGet packages need to be downloaded:

```bash
# Restore all packages defined in .csproj
dotnet restore
```

### Private NuGet Feeds

USACE uses a private NuGet feed (Nexus). To access it:

```bash
# Add the USACE Nexus source
dotnet nuget add source https://www.hec.usace.army.mil/nexus/repository/consequences-nuget-public/ --name USACENexus
```

## Development Workflow

### During Active Development (Project Reference)

When both the web app and library are changing frequently:

```xml
<!-- Use project reference for rapid iteration -->
<ItemGroup>
    <ProjectReference Include="..\..\System-Response\SystemResponse\SystemResponse.csproj" />
</ItemGroup>
```

Benefits:
- Edit library code, see changes immediately
- Single debugging session
- No version coordination needed

### For Stable Releases (NuGet Package)

When the library is stable and versioned:

```xml
<!-- Use NuGet for reproducible builds -->
<ItemGroup>
    <PackageReference Include="SystemResponse" Version="1.0.0" />
</ItemGroup>
```

Benefits:
- Reproducible builds across all machines
- Clear version dependencies
- No need for library repo locally

### Hybrid Approach (Common During Transition)

Keep both options available by commenting:

```xml
<ItemGroup>
    <!-- Uncomment ONE of these: -->

    <!-- For development (when modifying System-Response): -->
    <!-- <ProjectReference Include="..\..\System-Response\SystemResponse\SystemResponse.csproj" /> -->

    <!-- For production (use published package): -->
    <PackageReference Include="SystemResponse" Version="1.0.0" />
</ItemGroup>
```

### Version Ranges

You can specify version ranges in your .csproj:

```xml
<!-- Exact version -->
<PackageReference Include="SystemResponse" Version="1.0.0" />

<!-- Any 1.x version -->
<PackageReference Include="SystemResponse" Version="1.*" />

<!-- At least 1.0.0, less than 2.0.0 -->
<PackageReference Include="SystemResponse" Version="[1.0.0, 2.0.0)" />
```
