# System-Response Library

This guide explains the System-Response library architecture, particularly the distinction between using it as a **library** versus as an **API**, and how different applications can consume it.

---

## Library vs API: The Key Distinction

System-Response is fundamentally a **library** (like NumPy in Python)—not a web API. This distinction is critical for understanding how to use it correctly.

### What Is a Library?

A library is compiled code that other programs import and call directly:

```csharp
// Calling a library - direct function call, no network
using SystemResponse.Toolboxes.BEPProgression.Schmertmann;

var model = new SchmertmannModel(config);
var result = await model.RunAsync();  // Direct C# call
```

### What Is an API?

An API (in this context) is a web service that receives HTTP requests:

```csharp
// Calling an API - network request, HTTP overhead
var response = await httpClient.PostAsync(
    "http://calc-server/api/schmertmann/run",
    JsonContent.Create(config)
);
```

## Architecture Diagrams

### Option A: Direct Library Reference (Recommended)

This is how RMC-Toolboxes-WebApp uses System-Response:

```
┌─────────────────────────────────────────────────────────────────┐
│ Your Web App (RMC-Toolboxes-WebApp)                             │
│                                                                  │
│  RMCToolboxes.API                                               │
│  ├── Your own controllers                                       │
│  ├── Your own authentication                                    │
│  ├── Your own database                                          │
│  └── Directly calls: new SchmertmannModel(...).RunAsync()       │
│             │                                                    │
│             │ Direct C# call (no HTTP)                          │
│             ↓                                                    │
│  SystemResponse (Library) ←──── Project reference                │
└─────────────────────────────────────────────────────────────────┘

Pros:
✓ Faster (no HTTP overhead)
✓ Simpler (one process, not two)
✓ Full control (your auth, your database, your API shape)
✓ Type-safe (compile-time checking)
```

### Option B: HTTP Microservice (Alternative)

This is how you might use SystemResponse.API:

```
┌─────────────────────────────────────────────────────────────────┐
│ Your Web App                                                     │
│                                                                  │
│  Your API Layer                                                  │
│  └── Makes HTTP calls to: POST http://calc-server/api/...       │
│             │                                                    │
│             │ HTTP request over network                          │
│             ↓                                                    │
└─────────────────────────────────────────────────────────────────┘
             │
             │ HTTP (localhost or remote server)
             ↓
┌─────────────────────────────────────────────────────────────────┐
│ SystemResponse.API (Running as separate service)                 │
│                                                                  │
│  - Receives HTTP request                                         │
│  - Calls library: new OvertoppingAnalysis(...).RunAsync()       │
│  - Returns JSON response                                         │
└─────────────────────────────────────────────────────────────────┘

Pros:
✓ Can scale calculation server independently
✓ Can update calculations without touching web app
✓ Accessible from non-.NET clients

Cons:
✗ More complex (two services to deploy)
✗ Slower (HTTP overhead)
✗ Network failure handling needed
```

## Multiple Consumers

One of the main benefits of a library architecture is that multiple applications can use the same calculation engine.

### Multiple Web Applications

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ Web App A    │     │ Web App B    │     │ Web App C    │
│ (Toolboxes)  │     │ (Screening)  │     │ (Reports)    │
└──────┬───────┘     └──────┬───────┘     └──────┬───────┘
       │                    │                    │
       │ Project Reference  │ Project Reference  │ Project Reference
       └────────────────────┼────────────────────┘
                            ↓
              ┌─────────────────────────────┐
              │ System-Response LIBRARY     │
              │ (Shared by all)             │
              └─────────────────────────────┘
```

### Headless Applications

Command-line tools or batch processors can use the library directly:

```csharp
// Console application - no web server needed
using SystemResponse.Toolboxes.BEPProgression.Schmertmann;

class Program
{
    static async Task Main(string[] args)
    {
        // Load configuration from file
        var config = LoadConfigFromFile("input.json");

        // Run calculation directly
        var model = new SchmertmannModel(config);
        var result = await model.RunAsync();

        // Save results
        SaveResultsToFile(result, "output.json");

        Console.WriteLine($"Calculation complete: {result.Summary}");
    }
}
```

### Non-.NET Clients via API

If you have Python scripts that need calculations, they can call SystemResponse.API:

```python
import requests

# Python calling the System-Response API
response = requests.post(
    "http://calc-server:5000/api/overtopping/analyze",
    json={
        "embankmentHeight": 10.0,
        "downstreamSlope": 3.0,
        "baseDuration": 12,
        "embankmentErodibility": "erodible"
    }
)

result = response.json()
print(f"Fragility curve: {result['fragilityCurve']}")
```

## Should You Delete SystemResponse.API?

**No.** The API project is useful for several reasons:

1. **Reference Implementation**: Shows the correct way to expose calculations as REST endpoints
2. **Testing Tool**: Swagger UI allows quick testing without writing code
3. **Non-.NET Access**: Python scripts and other languages can use it
4. **Future Flexibility**: Microservice architecture option remains available

For your primary web application (RMC-Toolboxes-WebApp), you will:

1. **Ignore** SystemResponse.API—you don't need it
2. **Reference** SystemResponse (the library) directly
3. **Build** your own API layer in `backend-dotnet/RMCToolboxes.API`
