# PFM-01: Overtopping Erosion

PFM-01 (Potential Failure Mode 01) models overtopping erosion of embankment dams. This is the first fully implemented failure mode in the DST and serves as the template for all subsequent PFM modules.

Each PFM-01 location represents one embankment section being evaluated (e.g., "Main Embankment", "Left Abutment"). The module uses the **three-table SRP pattern**: a DST-calculated SRP, a user-defined SRP, and an active SRP that is synced from one of the two sources based on the user's selected mode.

## Three-Table SRP Pattern

The SRP (System Response Probability) pattern is central to PFM-01 and will be replicated across all future PFM modules. It uses three child tables per PFM location:

1. **DstSrp** — Stores the curves and nodal tables computed by the DST System-Response library. This is the "computed" result and is read-only from the user's perspective (though users can revise individual nodal estimates).

2. **UserSrp** — Stores a manually-entered SRP curve that the user computed outside DST using their own tools. This is fully user-defined.

3. **ActiveSrp** — A denormalized cache of whichever SRP is currently "active" based on the parent entity's `SrpMode` field. When the mode is `'dst'`, the active curve is synced from DstSrp's total SRP. When the mode is `'user'`, it's synced from UserSrp's curve.

Data is **never deleted** when switching modes. Switching from DST to User mode does not erase the DST calculations, and switching back does not erase user-entered curves. This preserves all work.

## Concurrency Handling

PFM-01 uses pessimistic locking for System-Response calculations. When a calculation is in progress, the `LastCalculatedAt` timestamp on `Pfm01_DstSrp` is checked to prevent concurrent overwrites.

## Entity Relationship Summary

```
Pfm01 (1) ──→ (0..1) Pfm01_DstSrp       [FK: Pfm01Id, UNIQUE, CASCADE DELETE]
Pfm01 (1) ──→ (0..1) Pfm01_UserSrp      [FK: Pfm01Id, UNIQUE, CASCADE DELETE]
Pfm01 (1) ──→ (0..1) Pfm01_ActiveSrp    [FK: Pfm01Id, UNIQUE, CASCADE DELETE]
```

All three child tables use a UNIQUE constraint on `Pfm01Id` to enforce the 1:1 relationship. CASCADE DELETE ensures that when a PFM-01 location is deleted, all associated SRP data is cleaned up automatically.
