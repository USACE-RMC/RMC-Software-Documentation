# Screening Management

The `Screenings` table is the top-level entity in the DST database. Each screening represents a complete dam screening assessment for one dam. All other data in the system — dam information, hazard characterization, failure mode analysis, consequences, and risk results — is associated with a single screening.

## Screening Codes

Each screening is assigned a short, unique alphanumeric code upon creation. This code serves as the URL-friendly identifier used in all routing within the application:

```
/screening/{Code}/dam-information
/screening/{Code}/pfm-01
/screening/{Code}/risk-analysis
```

Using codes instead of UUIDs in URLs makes them shorter and more readable, while the `UNIQUE` index on the `Code` column guarantees uniqueness.

## Future Plans

The Screenings table currently has no enforced foreign key relationships with downstream tables. The `ScreeningId` column in `Pfm01` (and future PFM tables) stores the screening association, but the FK constraint is not yet enforced at the database level. This will be added in a future migration to allow incremental development of each module.

The planned relationship structure:

```
Screenings (1) ──→ (N) Pfm01
Screenings (1) ──→ (N) Pfm02
...
Screenings (1) ──→ (N) Pfm33
Screenings (1) ──→ (1) DamOverview
Screenings (1) ──→ (N) HydrologicHazard
Screenings (1) ──→ (N) SeismicHazard
```
